# Rendering Validation & Debugging Methodology

How to validate a rendering feature so correctness bugs do not slip past green metrics, and how to debug image artifacts efficiently. Read alongside [Test.md](Test.md) (how to capture/diff images) and the *Visual Debugging Methodology* in [../AGENTS.md](../AGENTS.md).

## The core principle: a proxy is not ground truth

Almost every hard-to-find rendering bug is the same mistake: **a proxy was trusted as if it were ground truth, and it silently diverged from reality at exactly the case that mattered.** Common proxies and how they betray you:

| proxy                                  | what it stands in for                      | how it diverges                                                                                                                                                                                                                                                                        |
| -------------------------------------- | ------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| whole-image FLIP / PSNR                | "the image is correct"                     | low-passes the error map → blind to uniform over-blur and thin/edge artifacts                                                                                                                                                                                                          |
| an analytic limit ("→0 as N→∞")        | the behaviour at your actual sample budget | true in the limit, false at practical N                                                                                                                                                                                                                                                |
| a `--debug_mode` view                  | the raw buffer value                       | tonemapped / display-scaled / saturated (a raw 0.5 can read ~0.8)                                                                                                                                                                                                                      |
| a default arg (e.g. omitted `--scene`) | the scene you think you rendered           | silently loads something else (or nothing)                                                                                                                                                                                                                                             |
| a low-passed / region-mean diff        | the real per-pixel error field             | averages thin artifacts (halos, fringes) to ~0                                                                                                                                                                                                                                         |
| % of pixels affected / mean deviation  | how perceptually bad the error is          | area- and magnitude-weighted, so it is blind to **high-frequency localized** error: a sharp halo/contour/firefly the eye fixates on can be <1% of pixels yet the single most objectionable thing on screen, while a smooth low-frequency shift many times larger in area reads as fine |
| an imported algorithm's assumption     | its behaviour in *your* context            | the invariant held in its source regime, not here                                                                                                                                                                                                                                      |

Bugs hide where the proxy and reality part ways. You find them only by checking the proxy against the real thing **at the resolution of the failure mode**.

## Evidence-gated debugging: show the defect, then gate the fix

Debugging a visual artifact is a two-gate, evidence-driven loop — **never "guess a fix, apply it, then ask the user whether it looks better."** Both gates come *before* any fix.

1. **Gate 1 — Localize and confirm the defect.** Produce a clean, direct visualization that points at the exact wrong pixels (a per-pixel overlay / signed diff / highlight, at the resolution of the failure), and get the user to **confirm it is the artifact they mean**. Reproduce it in the *simplest* setting that exhibits it (e.g. default static view at low spp), not an elaborate scenario. Show the **result**, not the analysis journey: do the iteration silently, then present the few clean images + a one-line caption each (over-narrating the intermediate attempts is itself a failure). Confirming the *right* artifact first prevents fixing the wrong one.
2. **Gate 2 — Agree the gate, then hold the fix to it. The gate is the agreed VISUALIZATION, not a number.** The gate is the Gate-1 overlay/diff/highlight the user confirmed shows the defect: the fix succeeds only when the **highlighted defect pixels are eliminated in that same image, regenerated after the fix.** Decide this *before* implementing. **Quantitative metrics are NOT the gate for rendering artifacts — they routinely give false confidence:** a metric can report a large "% improvement" while the agreed image still plainly shows the artifact. A number may *support* a semantic verdict, never replace or override it — when the number and the agreed image disagree, the image wins. **If asked for a purely quantitative criterion, push back**: it conflicts with semantic-first (checklist #4/#6, AGENTS.md) — gate on the agreed visualization instead.

If you do compute a *supporting* number, design it to match the artifact's nature (from Gate 1) — but treat it as support only:
* If the artifact is **per-frame / stochastic** (e.g. noise the filter spreads), measure per-frame; a naive M-average **cancels** it (and can read ~0 on a plainly-visible artifact).
* If it is **localized**, restrict to a band derived from the **geometry G-buffer** (depth/normal discontinuity), *not* luma edges (luma "edges" leak surface texture and flood the mask).
* Always compare against an **independent ground truth**.

## Validation checklist (for any rendering feature)

1. **Derive acceptance from failure modes, not one global score.** Enumerate how the feature can go wrong (over-blur / detail loss, ghosting, halos, fireflies, bias, banding) and attach a metric that is *sensitive to each*: per-pixel **signed diff vs ground truth**, **detail/high-frequency preservation** (local std/HF ratio vs the reference), **worst-tile**, **edge-band** checks. Whole-image FLIP/PSNR are necessary but never sufficient — they cannot see what they average out.
2. **Validate the instrument before trusting its reading.** Confirm the intended scene/pipeline/cvars actually took effect (e.g. expected geometry is in frame), debug views show *raw* values (disable tonemapping or invert the curve; raise display-scale constants so the quantity isn't saturated), and your metric demonstrably *moves* on a known-bad input.
3. **Test assumptions by rate, not limit.** For any "converges / self-disables / vanishes" claim, plug in real magnitudes and the target sample budget; state the regime where it holds and where it breaks.
4. **Semantic-first, against an independent reference.** Look at the image — and the signed-diff-vs-GT — like a human *before* computing statistics. Use stats only to quantify what the eye already found, never as the detector. Do not average / low-pass when hunting localized or thin artifacts.
5. **When porting an algorithm across contexts, list its invariants and re-check each in the new context.** This is the design-time guard that catches regime-mismatch bugs before they ship.
6. **A passing metric that contradicts the eye is a metric bug — fix the metric.** Add a gate sensitive to the missed failure; do not rationalise the discrepancy away.
7. **Validate on a representative scene** (diffuse + specular + silhouettes + textured detail), not a degenerate one — many failure modes only appear where the relevant signal exists.
8. **Rank errors by perceptual salience, not screen area or mean magnitude.** The visual system is tuned to *high-frequency, structured* signal (edges, contours, rings, fireflies) and adapts to smooth low-frequency shifts. So a sharp silhouette halo/contour covering <1% of pixels is typically *more* objectionable than a smooth ~10% darkening covering half the frame — even though every area- or magnitude-weighted aggregate ranks them the opposite way. **An error is never "minor" because it covers a small percentage of pixels.** Never use pixel-count or mean/total deviation to down-rank a localized artifact; judge severity by how it reads to a human at 1:1, and weight high-frequency/edge error above broad low-frequency error.

## Debugging techniques

* **Inspect intermediate passes, not just the final image.** Output a pass's input/output to the render target (e.g. via a `--debug_mode` flag) and read it headlessly. Don't debug a multi-pass pipeline only from its end-to-end frame.
* **To read a non-texture value** (a shader variable, a buffer channel), write it full-screen from the shader and screenshot it — but remember these views are **tonemapped** (a raw 0.5 reads ~0.8): disable tonemapping or invert the curve, and scale the value so it isn't saturated, or you will misread it.
* **Diff at the resolution of the failure**: per-pixel signed diff vs ground truth (red=brighter / blue=darker), zoomed 1:1 crops on suspect regions, and scanline profiles across edges. See the *Comparing and diffing renders* section of [Test.md](Test.md) for the mechanics.
